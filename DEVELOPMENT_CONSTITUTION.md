# RiverWatch Development Constitution

## Principle 00 — Build Principles, Not Features
Do not change principles easily. When evidence changes, update the principles without hesitation.

## Principle 01 — Single Source of Truth
One information item has one owner. Duplicate display may be allowed; duplicate ownership is not.

## Principle 02 — Health vs Profile
Health describes current condition. Profile describes configuration. They must not be mixed.

## Principle 03 — Data over Code
Frequently changing values belong in data. Stable doctrine and architecture may remain in code.

## Principle 04 — Architecture First
Architecture is reviewed and approved before implementation.

## Principle 05 — Explain Before Predict
RiverWatch explains status before attempting prediction.

## Principle 06 — Keep the Boat Simple
New features must not increase complexity without clear operational benefit.

## Principle 07 — CAB Workflow
Backlog → Architecture Review → Sprint → Implementation → Sea Trial → Release.


## Principle 08 — Typography Baseline Lock
Typography is part of the product identity. Do not reduce font sizes or alter the typography scale unless explicitly approved through CAB.


## Principle 09 — Design Constitution Alignment
UI changes must preserve the Design Constitution. Typography, spacing, and component identity are managed design assets, not incidental CSS edits.


## CAB-020e Addendum
Intro screen typography is explicitly included in the ADR-002 Typography Baseline Lock scope.
